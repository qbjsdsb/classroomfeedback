# 课后反馈生成器 二期设计文档（Spec v2）

- **日期**：2026-06-26
- **状态**：待用户审阅
- **基于**：MVP（已实现，见 `2026-06-26-课后反馈生成器-design.md`）
- **范围**：核心增强 4 项。不含云同步+账号（架构级改造，留三期）、移动端录音适配（需接 ASR，留三期重谈）、图片/PDF 导出（独立低优先）、反馈打分（与修改差异学习重叠，合并到本期的"修改差异学习"）

---

## 1. 二期目标

在 MVP 单学生闭环之上，做 4 项核心增强，放大产品价值：

1. **多学生批量生成 + AI 拆分**：一次口述/输入整节课，AI 按学生姓名拆分，分别生成反馈
2. **多套规范档切换**：按科目维度管理规范档，选学生自动匹配其常用科目
3. **修改差异反向学习**：分析老师对 AI 输出的修改习惯，提议规范档优化
4. **学生画像 AI 自动回填**：从历史反馈提炼学生特征，提议补充档案

**核心安全原则**（贯穿 4 项）：
- 涉及"改规范档"或"改学生档案"的，**绝不自动执行**，一律手动触发 + AI 提议 + 老师逐条确认
- 涉及"自动判断"的（如 AI 拆分），**必经老师确认步骤**，错了可手动改
- 老师手填的数据**只追加不覆盖**

## 2. 功能一：多学生批量生成 + AI 拆分

### 2.1 交互流程

```
1. 生成页切换"单学生"/"批量"模式
2. 批量模式：勾选 N 个学生 + 选 1 个规范档 + 选科目
3. 老师点名口述/输入整节课内容：
   "张三今天学了幂函数，掌握不错但粗心；李四基础弱，幂函数概念没吃透，需补..."
4. [AI 调用 A] 按学生姓名拆分文本 → 输出 {学生名: 对应内容片段}
5. 拆分确认页：列出每个学生 + 其内容片段，老师可手动调整任一片段
   - 若某学生未被识别到 → 该学生内容框为空，老师手动补
   - 若 AI 把内容分错学生 → 老师直接改文本框
6. 老师确认拆分 → 点"批量生成"
7. [AI 调用 B] 串行循环：对每个学生用规范档+画像+历史反馈+其内容片段生成反馈
   - 进度提示："正在生成第 2/5 个…"
8. 结果页：每个学生一个可编辑预览卡片
   - 单个可"重新生成"
   - 单个失败可"重试"，不影响已成功的
   - 全部确认后"批量保存"
```

### 2.2 设计要点

- **拆分 prompt 关键**：把所选学生姓名列表给 AI，要求按姓名切分，输出 JSON `{"张三":"...","李四":"..."}`。未提及的学生返回空字符串。
- **拆分确认是硬步骤**：不跳过。这是降低"张冠李戴"的关键兜底。
- **串行生成**：延续 MVP 决策，不并行。N 个学生依次出，进度可见。
- **批量上限**：不设硬上限，但 UI 提示"超过 5 个学生会较慢，建议分批"。
- **部分失败处理**（落实 spec 8 节 C1/C2）：失败学生标记"重试"按钮，成功学生反馈先暂存不清空。
- **批量保存**：一次保存 N 条反馈，每条独立记录 aiOriginal 和 finalText。

### 2.3 与 MVP 的关系

- 复用 MVP 的 generateFeedback（单学生生成逻辑不变）
- 复用 useRecording、纠错流程（批量也支持先纠错再拆分）
- 新增"拆分"AI 调用 + 拆分确认页 + 批量结果页

## 3. 功能二：多套规范档切换

### 3.1 设计

MVP 已支持创建多个规范档（Task 17），但生成页是"独立选一个"，没有科目维度匹配。二期做轻量增强：

- **学生加"常用科目"字段**（可选）：学生档案增加 `defaultSubject` 字段
- **生成页自动筛选**：选学生后，规范档下拉自动筛选该科目的规范档，老师可手动改选
- **规范档的 subject 字段**：MVP 已有，二期强调按 subject 分组管理
- **规范档页按科目分组展示**：列表按 subject 分组，便于管理

### 3.2 不做的

- 不做"学生绑定特定规范档"（太碎，老师维护不过来）
- 不做自动切换（只筛选不强制）

## 4. 功能三：修改差异反向学习

### 4.1 触发机制（手动 + 提醒）

- MVP 已攒 feedbacks 表的 aiOriginal 和 finalText
- 当某规范档关联的、有修改差异的反馈攒够 **20 条**，规范档页提示"已积累 N 条修改记录，可分析您的修改习惯"
- 老师点"分析修改习惯" → [AI 调用] 分析最近 50 条差异 → 输出"修改建议清单"
- **绝不自动改规范档**

### 4.2 修改建议清单形态

AI 输出 JSON 数组，每条建议：
```json
{
  "suggestions": [
    {
      "field": "opening",
      "current": "该生今日",
      "observed": "老师常把'该生今日'改成学生名字开头",
      "proposal": "建议开头改为'【学生姓名】今日'",
      "evidenceCount": 12
    }
  ]
}
```

老师对每条建议：
- **采纳**：应用到规范档对应字段
- **拒绝**：忽略
- **锁定**：采纳并锁定该字段（防止下次学习再改）

### 4.3 设计要点

- 分析样本：取该规范档下、includeInLearning=true 且 aiOriginal≠finalText 的反馈，最近 50 条
- 不足 20 条不允许触发（提示"积累更多修改后再分析"）
- 采纳只改未锁定字段，锁定字段不受影响（与 relearn 一致）
- 每次分析是独立的，不累积"上次建议"

## 5. 功能四：学生画像 AI 自动回填

### 5.1 触发机制（手动 + 提议确认）

- 学生详情页加按钮"从反馈提炼特征"
- [AI 调用] 分析该学生最近 10 条反馈 → 输出"特征提议"
- **绝不自动覆盖老师手填内容**，只提议追加/补充

### 5.2 特征提议形态

AI 输出 JSON：
```json
{
  "weaknessesProposal": "根据反馈，该生在幂函数、二次函数等代数运算上反复出现粗心错误",
  "personalityProposal": "反馈中多次提到专注度波动，状态好时掌握快",
  "parentFocusProposal": "",
  "sourceFeedbackCount": 8
}
```

老师看到后：
- 每个字段旁边显示"当前值"和"AI 提议"
- 老师选择：**采纳追加**（提议内容追加到现有内容后）/ **替换**（用提议覆盖）/ **忽略**
- 空提议（`""`）表示 AI 没发现可补充的，不显示该字段

### 5.3 设计要点

- 只分析 includeInLearning=true 的反馈（避免老师明确不纳入学习的污染）
- 不足 3 条反馈不允许触发
- "年级"字段不参与 AI 回填（结构化事实，AI 不该改）
- 提议只针对 weaknesses/personality/parentFocus 三个自由文本字段

## 6. 数据模型变更

### 6.1 Student 表加字段

```ts
interface Student {
  // ...原有字段
  defaultSubject: string;  // 新增：常用科目，可选，用于规范档筛选
}
```

需 Dexie schema version 2 迁移。

### 6.2 新增 Suggestion 表（修改差异学习的建议记录，便于追溯）

```ts
interface Suggestion {
  id?: number;
  specProfileId: number;
  type: "style";  // 预留扩展
  field: string;
  current: string;
  proposal: string;
  observed: string;
  evidenceCount: number;
  status: "pending" | "accepted" | "rejected";  // 锁定通过 specProfile.lockedFields 实现，不在此表
  createdAt: number;
}
```

### 6.3 新增 ProfileSuggestion 表（学生画像提议，临时态，采纳后可删）

实际上画像提议是即用即弃的，不必持久化。AI 返回后直接在 UI 展示，老师处理完即丢。**不建表**，避免数据堆积。

## 7. AI 调用新增

| 调用 | callType | 用途 | 输入 | 输出 |
|---|---|---|---|---|
| split | 新增 `split` | 批量拆分 | 学生姓名列表 + 整段文本 | `{姓名: 片段}` JSON |
| analyzeEdits | 新增 `analyzeEdits` | 修改差异学习 | 50 条 (aiOriginal, finalText) 对 + 规范档 | 建议清单 JSON 数组 |
| extractProfile | 新增 `extractProfile` | 画像回填 | 学生最近 10 条反馈 | 3 字段提议 JSON |

CallType 类型扩展为 `"correct" | "generate" | "learn" | "split" | "analyzeEdits" | "extractProfile"`。

## 8. 风险与边界

| 风险 | 应对 |
|---|---|
| AI 拆分张冠李戴 | 拆分确认页是硬步骤，老师手动调整 |
| AI 拆分遗漏学生 | 未识别学生内容框为空，老师手动补 |
| 批量生成部分失败 | 单个重试，成功的不清空 |
| 修改建议改坏规范档 | 绝不自动改；逐条采纳；锁定字段不受影响 |
| 画像提议覆盖老师手填 | 绝不自动覆盖；提议形态，老师选追加/替换/忽略 |
| 批量生成 token 消耗大 | 进度提示 + 统计页可见；UI 提示超 5 个建议分批 |
| 拆分 JSON 解析失败 | 复用 MVP 的 parseJsonLoose 容错 |

## 9. 验收标准（二期完成定义）

1. 批量模式：选 3+ 学生 → 点名口述 → AI 拆分 → 确认拆分（可改）→ 批量生成 → 逐个预览编辑 → 单个可重试/重新生成 → 批量保存
2. 多套规范档：学生设常用科目后，生成页选学生自动筛选该科目规范档
3. 修改差异学习：攒 20+ 条差异 → 规范档页提示 → 分析 → 建议清单 → 逐条采纳/拒绝/锁定 → 规范档更新
4. 画像回填：学生有 3+ 条反馈 → 详情页点提炼 → 提议 → 采纳追加/替换/忽略 → 档案更新
5. 所有新功能不破坏 MVP 既有流程（单学生生成、导出导入、统计等回归通过）
6. 全量测试通过，tsc 无错，build 成功
