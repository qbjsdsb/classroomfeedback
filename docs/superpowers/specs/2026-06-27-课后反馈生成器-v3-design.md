## 1. 三期目标（交互体验层）

在二期视觉一致性之上，补齐交互反馈层，让产品"好用"而非仅"好看"：

1. **Toast 通知系统**：替换全部 alert，统一成功/错误/进度提示，分型时长 + 手动关闭
2. **Modal 确认对话框**：替换 confirm，破坏性操作用自定义弹窗
3. **Loading 全覆盖**：AI 调用复用 Toast loading 机制；页面初始加载用骨架屏
4. **引导式空状态**：列表为空时给图标 + 文案 + 引导操作按钮

**核心原则**：
- 零架构改动（纯前端 React，不动数据模型、不动路由）
- 零功能回归（32 测试全过）
- 零新依赖（自建组件，不引第三方库）

## 3. 架构设计

### 3.1 新增文件结构

```
src/
  components/
    NotificationProvider.tsx   # Context + Provider + Toast 容器 + Modal 容器
    Skeleton.tsx               # 骨架屏组件
    EmptyState.tsx             # 空状态组件
  hooks/
    useNotify.ts               # useContext 封装，暴露 notify API
tests/
  components/
    NotificationProvider.test.tsx   # TDD 测试
```

### 3.2 NotificationProvider 内部结构（单文件，约 150 行）

**Context 值：** `{ toast, confirm }` 两套 API

**状态：**
- `toasts: ToastItem[]` — Toast 队列
- `modal: ModalState | null` — 当前 Modal（同时只允许一个）

**ToastItem 类型：**
```ts
interface ToastItem {
  id: string;          // 自增或随机
  type: "success" | "error" | "info";
  message: string;
  duration: number;    // 0 = 不自动消失
}
```

**ModalState 类型：**
```ts
interface ModalState {
  title: string;
  message?: string;
  resolve: (ok: boolean) => void;
}
```

**渲染：**
- Toast 容器：固定右上角（`fixed top-4 right-4 z-50`），最多堆叠 4 个，超过自动清最早的
- Modal 容器：居中 + 半透明遮罩（`fixed inset-0 z-40 bg-black/40 flex items-center justify-center`）
- 两者都用 React Portal 渲染到 document.body

### 3.3 组件依赖关系

```
App.tsx
  └─ NotificationProvider（最外层包裹）
       └─ 原有路由树
            └─ 各页面通过 useNotify() 调用 notify.xxx
```

所有页面在 `useNotify()` 调用时，Provider 必须已在祖先节点。开发期若未包裹，`useNotify` throw 友好错误立即暴露。

## 4. Toast API 与行为

### 4.1 API 签名

```tsx
notify.success(message: string): string   // 返回 toast id
notify.error(message: string): string
notify.info(message: string, opts?: { duration?: number }): string  // duration 默认 3000，0=不消失
notify.dismiss(id?: string): void         // 无 id 清全部
```

### 4.2 Toast 类型与时长

| 类型 | 视觉 | 自动消失 | 手动关闭 | 用途 |
|------|------|---------|---------|------|
| success | 绿色左边框 + ✓ 图标 | 3s | ✕ 按钮 | 保存成功、导入完成、批量保存完成 |
| error | 红色左边框 + ✕ 图标 | 5s | ✕ 按钮 | 生成失败、纠错失败、拆分失败 |
| info | 蓝色左边框 + ⓘ 图标 | 3s（默认）/ 0=不消失 | ✕ 按钮 | 进度提示、分析中、提炼中 |

### 4.3 Loading 复用模式（关键设计）

AI 调用场景的标准用法：
```tsx
const tid = notify.info("生成中…", { duration: 0 });  // 不自动消失
try {
  await generateFeedback(...);
  notify.dismiss(tid);            // 清 loading
  notify.success("生成完成");
} catch (e) {
  notify.dismiss(tid);
  notify.error("失败：" + e.message);
}
```

Loading 提示复用 Toast 机制，无需第三套 spinner 组件。

**按钮文字 loading 保留**：按钮 disabled + 文字切换（如"生成中…"）仍有价值（按钮本身是操作入口，需禁用防重复点击）。Toast loading 与按钮文字 loading 并存。

### 4.4 Toast 容器行为

- 位置：`fixed top-4 right-4 z-50`
- 队列上限：4 个，超过自动清最早的
- 动画：从右侧滑入（`translate-x-full` → `translate-x-0`），150ms
- 手动关闭：每个 Toast 右上角 ✕ 按钮

## 5. Modal confirm API 与行为

### 5.1 API 签名

```tsx
notify.confirm(title: string, message?: string): Promise<boolean>
// 返回 true（确认）/ false（取消/点遮罩/按 ESC）
```

### 5.2 行为

- 调用后 Modal 居中弹出，背景半透明遮罩（`bg-black/40`）
- 两个按钮：「取消」（btn-soft 灰底）+「确认」（btn-danger 红底，破坏性操作语义）
- 关闭路径（均 resolve(false)）：点遮罩 / 按 ESC / 点取消
- 确认路径（resolve(true)）：点确认
- 同一时间只允许一个 Modal

### 5.3 alert/confirm 替换映射

| 文件 | 原调用 | 替换为 |
|------|--------|--------|
| StudentsPage.tsx | `confirm("确认删除该学生？")` | `await notify.confirm("删除学生", "确认删除该学生？此操作不可撤销。")` |
| SettingsPage.tsx | `alert("导入完成，请刷新页面")` | `notify.success("导入完成，请刷新页面")` |
| BatchGeneratePage.tsx | `alert("已保存 " + count + " 条反馈")` | `notify.success("已保存 " + count + " 条反馈")` |
| App.tsx | `alert("建议导出备份数据…")` | `notify.info("建议导出备份数据以防丢失（设置 → 导出）")` |

### 5.4 Modal 视觉

- 卡片样式复用 `.card`，宽度 `max-w-sm`，居中
- 标题加粗，message 灰字小号（可选）
- 按钮区右对齐

### 6.1 API


### 6.2 视觉

- 容器：`.card` 样式
- 每行：`<div className="h-4 bg-gray-200 rounded animate-pulse mb-2" />`

### 6.3 用途

## 7. EmptyState 空状态组件

### 7.1 API

```tsx
<EmptyState
  title="暂无学生"
  hint="添加你的第一个学生档案"
  action={<button className="btn-primary" onClick={startNew}>新建学生</button>}
/>
```

action 是 ReactNode，可以是 Link（路由跳转）或 button（组件内函数），由调用方决定。

### 7.2 视觉

- 居中布局
- 图标（emoji，大号灰字）+ 标题（灰字加粗）+ 提示（小灰字）+ 操作按钮
- 容器用 `.card` 但内容居中

### 7.3 用途与引导动作

| 场景 | 标题 | 提示 | 引导按钮 |
|------|------|------|---------|
| StudentDetailPage 暂无反馈 | "暂无历史反馈" | "为该学生生成第一条反馈" | Link → `/generate` |
| StatsPage 暂无数据 | "暂无消耗数据" | "生成反馈后这里会显示 Token 消耗" | Link → `/generate` |
| StudentsPage 学生列表为空 | "暂无学生" | "添加你的第一个学生档案" | `<button onClick={startNew}>新建学生</button>`（非 Link，因 startNew 是组件内函数） |

### 7.4 不做的

- 不为每个页面的每个子区块做空状态（如段落为空、样本为空），只覆盖主列表为空的情况
- SpecProfilePage 规范档为空（内置档已 seed，理论上不为空，不加）

### 9.1 NotificationProvider 测试用例

1. `notify.success(msg)` 渲染绿色 Toast，3s 后自动消失
2. `notify.error(msg)` 渲染红色 Toast，5s 后自动消失
3. `notify.info(msg, { duration: 0 })` 不自动消失，需手动 dismiss
4. `notify.dismiss(id)` 清指定，`notify.dismiss()` 清全部
5. `notify.confirm(title, msg)` 弹 Modal，点确认 resolve(true)
6. 点 Modal 遮罩 resolve(false)
7. Toast 超过 4 个自动清最早的
8. useNotify 未在 Provider 内 throw 错误

## 10. 风险与边界

| 风险 | 应对 |
|------|------|
| Provider 未包裹时 useNotify 返回 undefined | useNotify 内 throw 友好错误，开发期立即暴露 |
| Toast 队列堆积 | 最多 4 个，超过清最早 |
| Modal 与 Toast 同时存在 z-index 冲突 | Modal 遮罩 z-40，Toast z-50（Toast 在 Modal 之上，因 Toast 可能是 Modal 触发的反馈） |
| fake timer 测试自动消失 | 测试用 `vi.useFakeTimers()` + `vi.advanceTimersByTime()` |
| 改造面广导致回归 | 每个 AI 调用点的 loading 包裹严格 try/finally/dismiss 三段式；全量测试 + tsc + build 验收 |
| 内联错误与 Toast 错误重复 | 内联错误保留（就近提示），Toast 错误作为全局汇总，两者并存不冲突 |
