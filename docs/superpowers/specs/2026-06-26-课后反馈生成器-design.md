# 课后反馈生成器 — 设计文档（Spec）

- **日期**：2026-06-26
- **状态**：待用户审阅
- **作者**：brainstorming 协作产出

---

## 1. 概述

一个面向**单个教培老师自用**的课后反馈生成工具。老师通过录音或手动输入记录一节课的内容，AI 依据"从老师历史反馈中学习到的格式规范"生成贴合该学生情况的课后反馈。

核心差异化：**不预设固定模板，而是让 AI 从老师上传的历史反馈中分析归纳出"格式规范档"**，使产品能适应每家机构、每位老师的不同反馈风格。老师越用，AI 越懂他的写法。

---

## 2. 目标与非目标

### 2.1 目标（MVP 要达成）

- 老师能创建学生档案并维护核心画像字段
- 老师能通过录音（Web Speech API）或手动输入记录一节课内容
- 录音转写文本经 AI 纠错 + 老师校对后进入生成
- AI 能从老师上传的历史反馈中学习格式规范，产出可编辑的"规范档"
- 单学生场景下生成贴合学生情况、符合规范档风格的反馈
- 老师能预览、编辑、重新生成反馈，并一键复制或批量导出
- 全程数据本地存储（IndexedDB），支持导出导入备份
- 透明展示 Token 消耗

### 2.2 非目标（MVP 明确不做）

- 多老师 / 多机构 / SaaS 多租户
- 机构级权限管理、计费体系
- 课中实时录音整节课转写（重型 ASR）
- 多学生批量生成 + AI 自动拆分（二期）
- 多套规范档切换（二期）
- 学生画像 AI 自动回填（二期）
- 云同步 + 账号体系（二期）
- 反馈打分 / 反例学习（二期）
- 移动端录音适配优化（移动端仅保留手动输入）

## 3. 用户与使用场景

**唯一用户**：单个教培老师，自用。

**典型场景**：
1. 老师上完一节课，打开网站
2. 选择学生 + 科目
3. 对着电脑口述本节课情况（或手动输入）
4. AI 纠错 → 老师校对关键名 → 生成反馈
5. 预览反馈，不满意可重新生成或手动微调
6. 一键复制到微信发给家长，或批量导出存档
7. 累积一段时间后，上传更多历史反馈，手动触发"重新分析"让规范档更准

**新用户冷启动场景**（重点）：
- 新老师无历史反馈，若直接生成会得到"通用 AI 默认风格"，体验断裂
- 解决：内置 2~3 套示例规范档，新老师选一套即可立刻生成反馈，用顺手后再上传自己的历史反馈让 AI 学习

## 4. 核心概念与数据模型

### 4.1 核心实体

| 实体 | 说明 |
|---|---|
| **学生（Student）** | 老师管理的对象，含核心画像字段 + 历史反馈列表 |
| **规范档（SpecProfile）** | AI 从历史反馈学到的格式规范，按"老师 + 科目"维度，C 形态 |
| **历史反馈样本（HistorySample）** | 老师上传的过去反馈，用于学习规范档 |
| **生成的反馈（Feedback）** | 系统为某学生某节课产出的反馈，含 AI 原文与老师终版 |
| **修改差异记录（EditDiff）** | AI 原文 vs 老师终版的差异，MVP 只攒不用 |
| **Token 统计（TokenUsage）** | 每次 AI 调用的 token 消耗记录 |

### 4.2 规范档的"C 形态"

规范档 = **结构化骨架字段** + **每段附带自由说明**：

```
规范档
├─ 整体风格
│   ├─ 语气：[正式书面 / 半书面 / 口语]（字段）
│   └─ 风格自由说明：（自由文本，如"亲切但不啰嗦，多用肯定句"）
├─ 段落数：N（字段）
├─ 每段：
│   ├─ 标题：（字段）
│   ├─ 目标字数：（字段）
│   ├─ 内容要点：（字段）
│   └─ 段落自由说明：（自由文本）
└─ 常用开头 / 结尾：（字段 + 自由说明）
```

老师可见可改。AI 分析历史反馈后产出此结构，老师可在表单中编辑任一字段或自由说明。

### 4.3 学生画像（乙的轻量版）

只让老师手填 3~5 个关键字段，不追求大而全：

- 年级
- 性格特点（自由文本，一两句）
- 薄弱点（自由文本）
- 家长关注点（自由文本）

不做 AI 自动回填（二期）。

## 5. 功能规格

### 5.1 学生管理

- 增删改查学生
- 每个学生维护 4 个画像字段（见 4.3）
- 点进单个学生可查看其历史反馈列表
- 删除学生前确认，避免误删

### 5.2 规范档管理

**学习机制（核心）**：
- 老师上传历史反馈样本（支持 Word/TXT/Excel/粘贴，AI 自判格式）
- 点击"重新分析" → 系统取**最近 50 条**历史反馈全量重算 → AI 产出规范档
- 全量重算：每次基于完整样本重跑，不存在增量漂移
- **手动触发 + 提示**：上传后提示"您新上传了 N 条反馈，建议重新分析 [立即更新]"，不自动跑
- 老师可**锁定**规范档中某些字段为"硬约束"，重算时锁定字段不被改写

**编辑机制**：
- 规范档以 C 形态展示（结构化字段 + 自由说明）
- 老师可编辑任一字段
- 编辑后立即生效，下次生成即用新规范档

**内置示例规范档（冷启动）**：
- 系统预置 2~3 套示例规范档（如"简短口语风""正式书面风""详细分段风"），每套带 3~5 条示例反馈
- 新老师首次进入选一套作为起点，立刻可生成
- 后续可切换为自己的规范档

### 5.3 反馈生成（MVP 单学生）

完整链路：

```
1. 老师选 1 个学生 + 科目
2. 录音（Web Speech API 实时转写）或 手动输入（带快捷模板占位符）
3. 文本进入编辑框
4. [AI 调用] 文本纠错（结合上下文纠正常见错别字，尤其人名/术语）
5. 老师在编辑框校对关键名（重点确认人名、学科术语）
6. 老师点"生成反馈"
7. [AI 调用] 拿 规范档 + 学生画像 + 本节课内容 + 1~2 条该学生历史反馈 → 生成反馈
8. 预览反馈（可编辑、可"重新生成"）
9. 老师确认保存 → 存档 + 记录 AI 原文与终版差异
```

**快捷模板**：手动输入时提供占位符按钮，如【学生姓名】【今日知识点】【课堂表现】【家庭建议】，点击插入到光标处，降低面对空白框的压力。

### 5.4 修改差异记录（MVP 只攒不用）

- 每次生成反馈，同时保存"AI 原始输出"和"老师最终保存版本"
- MVP 不用于学习，仅累积存储
- 二期做差异分析反向修正规范档

### 5.5 复制与导出（闭环"发出去"）

- **一键复制**：单条反馈点"复制"进剪贴板，直接粘到微信
- **批量导出**：多条反馈导出为 TXT（按学生分块）
- 图片导出为二期

### 5.6 Token 统计

- 每次 AI 调用记录消耗 token
- 统计界面展示：总消耗、按调用类型分布（纠错/生成/学习）、按日期趋势
- Token 统计纳入导出导入备份

### 5.7 数据备份

- **导出**：一键导出全部数据为 JSON 文件，明文，导出按钮旁警示"此文件含学生信息，请妥善保管"
- **导入**：上传 JSON 恢复
- **备份提醒**：首次创建学生、积累满 10 条反馈、检测到 7 天未备份时，弹窗提醒导出
- 二期加云同步 + 账号

## 6. AI 调用架构

### 6.1 模型选择

- **全用 DeepSeek V3**（快、便宜，适合确定性任务）
- MVP 先用 V3 跑通全链路，质量不够再考虑学习任务换 R1

### 6.2 调用链与串行策略

单次生成反馈的 AI 调用链：

```
[调用1] 文本纠错（1 次，整段）
[调用2] 生成反馈（1 次，单学生）
   └─ 老师点"重新生成" → 再调 1 次
```

- **串行执行 + 进度提示**（"正在生成…"），不并行
- 每次调用独立记录 token

### 6.3 规范档学习的调用

- 全量重算：取最近 50 条历史反馈，**一次性**丢给 AI 分析产出规范档（不分批，避免合并丢细节）
- 单次大调用，接受 token 较大、稍慢

### 6.4 Prompt 模板管理

- **写死在前端代码**，prompt 是开发者职责，不暴露给老师编辑
- 修改 prompt 需重新部署

### 6.5 API Key 管理

- 老师自带 DeepSeek API Key，首次使用在设置中粘贴
- Key 存 IndexedDB，调用时直接前端发请求给 DeepSeek
- **不写死开发者 Key 在前端**（会被盗刷）
- ⚠️ **未决风险**：DeepSeek API 是否允许浏览器跨域（CORS）调用，需在实现前实测验证。若不允许，需加 Cloudflare Worker 轻后端代理（仍免费），GitHub Pages 纯静态方案需调整为"前端 + Worker 代理"。

## 7. 技术架构

### 7.1 部署

- **GitHub Pages** 纯静态托管（HTML/CSS/JS）
- 数据不存 GitHub Pages，存用户浏览器 IndexedDB
- 零成本、零运维

### 7.2 存储

- **IndexedDB** 存所有动态数据（学生、规范档、历史反馈、生成的反馈、修改差异、token 统计）
- 不用 localStorage（容量小、不适合结构化数据）
- ⚠️ IndexedDB 有清空风险（清缓存、换设备、浏览器自动清理），靠导出导入备份兜底（见 5.7）

### 7.3 录音与移动端策略

- **录音仅电脑端**：检测设备，电脑端显示录音按钮，移动端隐藏录音只留手动输入
- Web Speech API 在 HTTPS（GitHub Pages 满足）下可用
- ⚠️ Web Speech API 中文识别质量一般，专有名词易错，靠 5.3 第 4~5 步纠错+校对兜底
- ⚠️ iOS Safari、微信内置浏览器录音不可靠，故移动端不暴露录音

### 7.4 技术栈（建议，实现阶段可调）

- 前端框架：React 或 Vue（实现阶段定）
- 构建：Vite
- 存储：IndexedDB 封装（如 Dexie.js）
- 部署：GitHub Pages

## 8. 错误处理与兜底

| 场景 | 处理 |
|---|---|
| AI 调用失败（网络/Key 失效/欠费/限流） | 自动重试 1~2 次；仍失败提示"生成失败，可重试或手动编写" |
| 老师输入的课堂内容 | 草稿自动保存到 IndexedDB，刷新不丢 |
| 生成质量不行 | 预览必经编辑（B1）；提供"重新生成"按钮（B2） |
| 批量部分失败（二期） | 失败可单独重试（C1），成功的不清空（C2） |
| IndexedDB 数据丢失 | 导出导入备份（5.7）；二期云同步 |
| 二次编辑已存档反馈 | 可编辑；老师自主勾选"是否纳入学习库"，避免污染 |

**MVP 保留**：A1 重试、A2 草稿保存、B1 预览编辑、B2 重新生成。
**MVP 不做**：A3 离线降级（断网场景少，不过度设计）、B3 反馈打分/反例学习（二期）。

## 12. 风险与未决事项

| 风险 | 影响 | 应对 |
|---|---|---|
| **DeepSeek API CORS 未知** | 可能阻断纯前端方案 | 实现前实测；若不允许改用 Cloudflare Worker 代理 |
| Web Speech API 中文质量 | 反馈错别字多 | 纠错 + 老师校对兜底；接受 tradeoff |
| IndexedDB 数据丢失 | 老师数据没了不可挽回 | 导出导入 + 备份提醒；二期云同步 |
| iOS / 微信浏览器录音不可靠 | 移动端录音不可用 | 移动端隐藏录音，只留手动输入 |
| AI 生成质量不达标 | 老师弃用 | 规范档学习 + 预览编辑 + 重新生成兜底 |
| 老师自带 Key 有门槛 | 转化率低 | 设置引导清晰；二期可考虑代填方案 |

## 13. 验收标准（MVP 完成定义）

1. 新老师打开网站 → 选内置规范档 → 创建学生 → 录音/输入 → 生成出像样反馈
2. 老师上传 5+ 条历史反馈 → 重新分析 → 规范档更新 → 后续生成贴合该风格
3. 生成的反馈可预览、编辑、重新生成、一键复制、批量导出
4. Token 消耗可见
5. 数据可导出为 JSON 并能导入恢复
6. 移动端打开不报错，录音按钮不显示，手动输入可用
7. API 失败有重试和提示，不白屏
